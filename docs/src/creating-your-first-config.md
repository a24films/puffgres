# Creating Your First Config

The basic unit of puffgres is a **config**, which defines a mapping between a Postgres table and a turbopuffer namespace. Run `puffgres new {name}` to create one:

```sh
puffgres new internal_film
```

This generates a timestamped directory:

```
configs/1772230207731_film/
├── config.toml
└── transform.ts
```

After filling in your `config.toml`, run `puffgres generate` to pull your table's schema from Postgres and create `schema.ts` — the transform imports this to get typed column definitions.

The timestamp ensures configs are applied in creation order, like database migrations. Configs are immutable — both the config and transform are content-hashed, and puffgres will refuse to run if the code has changed after being applied (state is maintained in a separate SQLite DB). We enforce immutability to prevent a turbopuffer namespace from containing rows produced by two different versions of a transform. If you need to change how a config works, tombstone the old one and create a new one (see [Applying](#applying)) for more. 

## Configs

`config.toml` defines the source table and destination namespace:

```toml
name = "film"
namespace = "internal_film_title"

[source]
schema = "public"
table = "internal_film"

[id]
column = "id"
type = "string"
```

### Fields

- **`name`** — identifier for this config.
- **`namespace`** — the destination turbopuffer namespace. Must be unique across all configs; puffgres will error if two configs target the same namespace.
- **`source.schema`** — the Postgres schema. Right now, only `public` is supported.
- **`source.table`** — the Postgres table to replicate.
- **`id.column`** — the column used as the primary identifier. Must be a single column — composite keys are not supported.
- **`id.type`** — the type of the id column. One of: `uint`, `int`, `uuid`, `string`.


## Schema

`schema.ts` is auto-generated by `puffgres generate`, which reads your Postgres table and produces typed column definitions. You should not edit this file — it gets regenerated whenever you run `puffgres generate` (and puffgres will error if you use a non-generated file here).

```typescript
/* eslint-disable */
// prettier-ignore
// Auto-generated by `puffgres generate`. Do not edit.
// Source: public.smart_buyer

import { parseRow as parseRowInternal, parseRows as parseRowsInternal, type Column } from "../../utils/puffgres";

export const columns = [
  { name: "id", type: "string" },
  { name: "buyer_name", type: "string" },
  { name: "buyer_id", type: "string" },
] as const satisfies readonly Column[];

export const parseRow = (cols: (string | null)[]) => parseRowInternal(columns, cols);
export const parseRows = (rows: (string | null)[][]) => parseRowsInternal(columns, rows);
```

The generated `parseRow` function converts raw column arrays into typed objects, handling type coercion for `uint`, `int`, `float`, `bool`, `uuid`, `json`, and `string` values. This allows you to use typed transformations. 

## Transforms

Transforms are a long-lived TypeScript processes that communicates with puffgres over stdin/stdout using JSONL (newline-delimited JSON). You define them in a `transform.ts` file, and they let you manipulate data, for example tokenizing an input, embedding it, concatenating it with other fields, etc, before upserting. 


### Event

Each event represents a raw Postgres row change, and we use generated `parseRow` from `schema.ts` to convert `columns` into a typed object.

```typescript
interface Event {
  operation: "insert" | "update" | "delete";
  id: number | string;
  columns: (string | null)[];
}
```

### Action

Each action tells puffgres what to do in turbopuffer. Return one of:

```typescript
type Action =
  | {
      type: "upsert";
      id: number | string;
      document: Record<string, unknown>;
      vector?: number[];
      distance_metric?: string;
      schema?: Record<string, unknown>;
    }
  | { type: "delete"; id: number | string }
  | { type: "skip" };
```

- **`upsert`** — write a document to turbopuffer. `vector` and `distance_metric` (`"cosine_distance"` or `"euclidean_squared"`) are required together. `schema` defines attribute types for the namespace (see [turbopuffer docs](https://turbopuffer.com/docs/write#schema)).
- **`delete`** — remove the document with this id.
- **`skip`** — do nothing for this event.

### Example

Here's a complete transform that embeds buyer names and upserts them with cosine distance search:

```typescript
import "../../utils/load-env.ts";
import { createInterface } from "readline";
import { embedBatchZeroEntropy } from "../../utils/embed-zeroentropy.ts";
import { parseRow } from "./schema.ts";

interface Event {
  operation: "insert" | "update" | "delete";
  id: number | string;
  columns: (string | null)[];
}

type Action =
  | {
      type: "upsert";
      id: number | string;
      document: Record<string, unknown>;
      vector?: number[];
      distance_metric?: string;
      schema?: Record<string, unknown>;
    }
  | { type: "delete"; id: number | string }
  | { type: "skip" };

const rl = createInterface({ input: process.stdin });

for await (const line of rl) {
  const input: Event[] = JSON.parse(line);

  const upsertEvents = input.filter((e) => e.operation !== "delete");
  const buyerNames = upsertEvents.map(
    (e) => parseRow(e.columns).buyer_name ?? "",
  );
  const vectors = await embedBatchZeroEntropy(buyerNames);
  const vectorMap = new Map(upsertEvents.map((e, i) => [e.id, vectors[i]]));

  const output: Action[] = input.map((event) => {
    if (event.operation === "delete") {
      return { type: "delete", id: event.id };
    }

    const row = parseRow(event.columns);

    return {
      type: "upsert",
      id: event.id,
      document: {
        buyer_name: row.buyer_name,
      },
      vector: vectorMap.get(event.id)!,
      distance_metric: "cosine_distance",
      schema: {
        buyer_name: { type: "string" },
      },
    };
  });

  process.stdout.write(JSON.stringify(output) + "\n");
}
```

There's a full working version of this in `examples/buyer_name/`.

## Applying

Before `puffgres run` will pick up a config, you need to apply it with `puffgres apply`. The run command only loads configs that have already been applied to the state database.

### Testing locally

Before applying, you can validate your config and transform with:

```sh
puffgres dry-run
```

This pulls a row from the database as defined in the config, pushes it through your transform, and logs the output. This is nice for catching bugs early.

### Checking

```sh
puffgres check
```

Validates all configs against the live database without applying. This verifies that `schema.ts` files are up to date (run `puffgres generate` if they aren't), that the referenced tables exist, and that transforms run successfully. Good to run in CI.

### Applying configs

```sh
puffgres apply
```

This registers all configs into the SQLite state database so they're picked up for replication. The config and transform are content-hashed at this point — if you change either after applying, puffgres will error.

### Running

```sh
puffgres backfill    # run only the backfill
puffgres run         # run the full CDC loop (backfill first, then streaming)
```

### Tombstoning

If you want to stop replicating a config (e.g., to replace it with a new version), tombstone it:

```sh
puffgres tombstone --name {config_name}
```

This generates a `tombstone.toml` in the config's directory, and the CDC loop will skip it.

### Removing

To fully remove a config — deleting the turbopuffer namespace, clearing all state from SQLite, and removing the config directory:

```sh
puffgres remove {config_name}
```

To remove the most recently applied config:

```sh
puffgres remove --last
```

### Debugging

```sh
puffgres debug
```

Launches a web UI (default port 3333) for inspecting turbopuffer namespace contents and viewing the live replication stream.
